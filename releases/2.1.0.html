New Dedicated APIs & Unified Cache

Players, NPCs, Objects, GroundItems and PlayerState now each have a dedicated API, a cache and a unified interaction model. Working with NPCs should feel the same as working with players or objects.

Previous System: Two separate caching systems running simultaneously - the old system and a newer one introduced by Vox and George. While the newer system improved performance in some cases, it added significant complexity and maintenance overhead.

Breaking Change: This is a breaking change that will require updates if you want to continue using Microbot. Anyone who wants to keep using the old scheduler can branch off.

Why Remove the Scheduler?

‚ùå Problems with Old Scheduler
  • Too complex and spread across many places
  • Required custom code in every plugin
  • Did not align with simplicity philosophy
  • Difficult to maintain and debug
  • Created hidden dependencies
‚úÖ Benefits of Removal
  • Cleaner, more predictable codebase
  • Reduced complexity for developers
  • Allows rest of client to move forward
  • Paves way for better design in future
  • Aligns with project philosophy
Future Plans: The scheduler will be redesigned in the future with a simpler, more maintainable architecture. This removal is temporary to allow the core client to progress without being held back by legacy complexity.

Impact on Plugin Compatibility

~70% of plugins will require updates to work with v2.1.0

What This Means For You:

If you're currently using plugins that depend on the scheduler, they will need to be updated. Plugin developers are encouraged to refactor their code to work without the scheduler. For users who need the old functionality, you can continue using v2.0.x or create a custom branch.

Unified Profile System

The logic for worlds, breaks and login flow currently lives in several separate places. The plan is to move all of it into a single profile system. A profile will hold all settings such as playtimes, break rules and world preferences.

üìã Before: Scattered Configuration

  • Login settings in one location
  • Break rules in another
  • World preferences elsewhere
  • Playtimes configured separately

‚ú® After: Single Profile System

  • All settings in one profile

  • Pre-configured before botting starts
  • Smoother and more predictable experience
